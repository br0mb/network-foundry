# Sysadmin Practice Workflow — Building a "Dojo" for Real-World IT Skills

> **Purpose:** A structured practice environment where theoretical knowledge from degrees and certifications meets hands-on execution. Daily tasks modeled on real job descriptions, generated by an automated ticket system, and completed using documentation — the same way a junior sysadmin works at a real company.

---

## The Problem This Solves

Degrees and certifications teach you *what* things are. A BS in Cybersecurity teaches you that Active Directory uses LDAP, Kerberos, and DNS. A CCNA teaches you how VLANs and routing work. But neither teaches you *how to do the work* — which package to install, which config file to edit, which command to run, what to check when something breaks. That gap between theory and practice is what every graduate hits when they walk into their first IT job.

This workflow bridges that gap by creating a practice environment — a "dojo" — where you work through real infrastructure tasks on real hardware, following documentation, troubleshooting real failures, and building the muscle memory that turns knowledge into skill.

## The Concept

The lab environment (Network Foundry) is the infrastructure. The sysadmin workflow is the practice routine. Together they simulate what a junior-to-mid sysadmin does at a real company:

1. **Morning checkup** — verify all services are running, check logs, review security alerts
2. **Daily ticket** — an automated system generates a task modeled on a real job description
3. **Execution** — work through the task using reference documentation (the same way you'd use a runbook at a job)
4. **Troubleshooting** — when things break (and they do), diagnose and fix using the same methodology a sysadmin uses
5. **Documentation** — every change is documented so it can be reproduced

## How It Works

### Automated Ticket Generation

A Python script runs via cron at 2:00 PM daily (30 minutes before "shift" start). It generates a ticket file containing:

- A morning checkup checklist (service status, network health, client health, security review, log review)
- The day's assigned task with specific checkboxes and reference pointers
- An end-of-day summary template

The tickets follow a 4-week rotation that cycles through all major sysadmin skill areas:

| Week | Mon | Tue | Wed | Thu | Fri | Sat | Sun |
|------|-----|-----|-----|-----|-----|-----|-----|
| 1 | Documentation | User Management | DNS Management | File Services | Security Review | Monitoring | Lab Maintenance |
| 2 | Documentation | DHCP Management | Print Server | Backup Practice | VPN Management | Monitoring | Lab Maintenance |
| 3 | Documentation | Server Migration | Patch Management | Certificate Authority | Security Hardening | Monitoring | Lab Maintenance |
| 4 | Documentation | Disaster Recovery | Automation | Network Troubleshooting | Full Audit | Monitoring | Monthly Review |

Each task maps directly to real-world sysadmin responsibilities found in job descriptions for Systems Administrator and MSP IT Engineer roles.

### Documentation as Runbooks

The workflow includes reference documentation that serves as runbooks — the same concept as documentation at a real job. When you hit a wall during a task, you consult the reference file instead of asking for help. If the reference doesn't have the answer, you use `--help`, `man` pages, or search — the same escalation path a junior sysadmin uses at a job.

Reference files include:
- **Samba AD DC Command Reference** — user, group, DNS, computer, and service management commands with Windows equivalents
- **WireGuard VPN Lab Guide** — full VPN build with troubleshooting section covering 7 real issues encountered during deployment
- **Independent Lab Reference** — documentation for 10 progressive tasks from easy to hard
- **Domain Join Package Reference** — explanation of every package required for Linux AD integration and how they work together

### Progressive Difficulty

The workflow starts with easy tasks (reboot verification) and increases to hard tasks (DHCP server with relay routing, CUPS print server, disaster recovery). The progression mirrors how skills develop at a real job — you start with routine checks, move to user management and DNS, then advance to infrastructure builds and troubleshooting.

### Build Reports as Evidence

Every infrastructure build is documented as a report containing:
- What was built
- Why it was built (the problem it solves)
- Step-by-step configuration
- Troubleshooting issues encountered and their solutions
- Enterprise relevance mapping (how the lab task maps to real-world responsibilities)

These reports serve as portfolio evidence that demonstrates hands-on experience with real infrastructure work.

## What Was Built

The following infrastructure was designed, configured, and documented as part of this workflow:

### WireGuard VPN (Remote Access)
- WireGuard server on the domain controller with key-based authentication
- Split tunneling with AllowedIPs controlling traffic routing
- IP forwarding and NAT/MASQUERADE for tunnel-to-LAN routing
- UFW firewall rules for tunnel interface and VPN port
- Client configuration with DNS through the tunnel
- 7 real troubleshooting issues documented (key swap, UFW missing rule, DNS binding, systemd-resolved, WSL networking, DHCP IP change, route metrics)

### NAT Gateway
- iptables MASQUERADE rules for internet access from isolated lab network
- IP forwarding between lab and internet interfaces
- Persistent rules via iptables-persistent
- Client gateway configuration via netplan

### DHCP Server with Relay
- ISC DHCP server on the domain controller for VLAN 10
- Cisco router configured as DHCP relay (ip helper-address)
- R1's native DHCP pool removed for the target VLAN
- Static route added for inter-VLAN DHCP response routing
- Full DORA handshake verified through the relay chain

### Active Directory Domain Join (Linux Client)
- Ubuntu Desktop VM joined to Samba AD domain via realmd/sssd
- Kerberos authentication with TGT verification
- DNS service discovery via SRV records
- Home directory auto-creation via oddjob-mkhomedir
- systemd-resolved DNS configuration with search domain

### CUPS Print Server
- CUPS with virtual PDF printer (no physical printer needed)
- Web dashboard for print queue management
- Network-accessible printing from domain clients
- CUPS configuration for network listening and access control

### Samba File Shares
- Department-based shares with group access control (valid users)
- Personal home drives with individual user permissions
- Cross-department access denial verified
- File creation and ownership verification through AD UID mapping

## AI Collaboration Model

This workflow was developed through collaboration between a human engineer and an AI agent acting as a senior engineer/mentor. The model is:

1. **Human directs** — the engineer decides what to build, what problems to solve, what tasks to practice
2. **AI guides** — the AI agent provides step-by-step instructions, explains concepts, and troubleshoots issues
3. **Human executes** — the engineer types every command, reads every output, makes every decision
4. **Both document** — the AI writes reference documentation; the human verifies and adjusts

This mirrors the senior/junior engineer dynamic at a real company. The AI is not a replacement for the engineer — it's a mentor that makes knowledge accessible and provides guidance. The engineer does the work. The AI points the way.
